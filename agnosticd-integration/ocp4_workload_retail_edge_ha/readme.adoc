= ocp4_workload_retail_edge_ha - Retail Edge HA Workshop

== Role Overview

This AgnosticD workload role deploys the *Retail Edge HA Workshop* to an existing OpenShift 4.21+ cluster with OpenShift Virtualization. The workshop teaches students three different high-availability architectures commonly used in retail edge computing environments:

* *Module 1*: Two-node Pacemaker/Corosync cluster with STONITH fencing
* *Module 2*: MicroShift with VRRP-based virtual IP failover
* *Module 3*: Two-node OpenShift cluster with remote etcd arbiter

The role uses a *wrapper pattern* that orchestrates the existing Helm chart deployment rather than reimplementing resources in Ansible. This approach preserves the proven GitOps architecture while integrating with AgnosticD's lifecycle management.

== Architecture

The workshop deploys using:

* *GitOps*: ArgoCD App of Apps pattern (1 parent + 8 child applications)
* *Multi-user Isolation*: Dual namespaces per student (workload + UDN)
* *Networking*: User Defined Networks (UDNs) for Layer 2 connectivity
* *VMs*: 3-9 VirtualMachines per student (depending on modules enabled)
* *Lab Guide*: Showroom instances with embedded terminals

=== Resource Requirements

Per student (all modules enabled):

* *CPU*: 18 cores (2+2+2 for Module 1-2, 4+4+1 for Module 3)
* *Memory*: 36 GiB (4+4+4 for Module 1-2, 16+16+2 for Module 3)
* *Storage*: 200 GiB
* *Namespaces*: 2 (workload + UDN)

For 50 students (max): ~900 CPU cores, 1.8 TiB RAM, 10 TiB storage

== Prerequisites

=== Required Cluster Configuration

* OpenShift 4.21 or later
* OpenShift Virtualization operator installed
* Storage class with RBD support (or default storage class)
* OVN-Kubernetes CNI (for User Defined Networks)
* ArgoCD operator (installed by Helm chart if not present)

=== Required Tools in Execution Environment

* Helm 3.x (included in AgnosticD v2 EE)
* Git (included in AgnosticD v2 EE)
* kubernetes.core Ansible collection

== Role Structure and Deviations from Canonical AgnosticD Pattern

=== Task File Layout

The role `tasks/` directory contains the following files:

[cols="1,2",options="header"]
|===
| File | Purpose

| `main.yml`
| Canonical ACTION dispatcher — routes to one of the five task files below based on `ACTION`.

| `workload.yml`
| Provision tasks: calls `pre_workload.yml`, deploys the Helm chart, waits for ArgoCD sync, then calls `post_workload.yml`.

| `pre_workload.yml`
| Pre-provision validation: verifies OCP version, discovers cluster domain and API URL, auto-installs RHACM and CNV when required.

| `post_workload.yml`
| Post-provision validation and reporting: waits for student namespaces and Showroom pods, then writes all access URLs to `agnosticd_user_info`.

| `remove_workload.yml`
| Destroy tasks: uninstalls the Helm release and removes all student and Showroom namespaces.

| `stop_workload.yml`
| Stop lifecycle: sets `spec.running: false` on all student VirtualMachines.

| `start_workload.yml`
| Start lifecycle: sets `spec.running: true` on all student VirtualMachines and waits for Running state.

| `status_workload.yml`
| Status lifecycle: reports VM, Showroom, and ArgoCD state via `agnosticd_user_info`.

| `install_rhacm.yml`
| Helper (not dispatched from `main.yml`): installs the RHACM operator. Included by `pre_workload.yml` when RHACM is absent and auto-install is enabled.

| `install_virtualization.yml`
| Helper (not dispatched from `main.yml`): installs the OpenShift Virtualization operator. Included by `pre_workload.yml` when CNV is absent and auto-install is enabled.
|===

=== Intentional Deviations from Canonical Structure

The following intentional deviations from the AgnosticD v2 canonical workload pattern are documented here for RHDP catalog review:

. *`pre_workload.yml` and `post_workload.yml` as separate files* — The provision flow is split into three files (`pre_workload.yml` → `workload.yml` → `post_workload.yml`) to keep each file focused on a single concern (validation, deploy, and reporting). `workload.yml` includes both at its start and end via `include_tasks`.

. *Helper task files (`install_rhacm.yml`, `install_virtualization.yml`)* — These are included conditionally by `pre_workload.yml` and are never dispatched directly from `main.yml`. They exist as separate files because operator installation is a substantial sub-task with its own retry loops and is reused in different contexts.

. *No `bastion_*` keys in `agnosticd_user_info`* — This is a workload-on-existing-OCP role; there is no bastion host. Showroom's Wetty terminal connects to VMs inside the OpenShift cluster using `oc rsh` / `virtctl console`, not via external SSH. The `bastion_public_hostname`, `bastion_ssh_user_name`, and `bastion_ssh_password` keys are intentionally omitted and do not need to appear in `provision-user-data.yaml`.

. *Showroom namespace pattern* — Showroom namespaces follow `showroom-{{ guid }}-student{{ '%02d' | format(item) }}`, matching the AgnosticD v2 standard of `showroom-<guid>-<username>` where `<username>` matches the `user:` key passed to `agnosticd_user_info`.

== Cleanup

=== Remove Workshop

[source,bash]
----
./bin/agd destroy \
  --guid workshop-test \
  --config openshift-workloads \
  --account sandbox1234
----

This will:

1. Uninstall the Helm release
2. Delete ArgoCD applications (cascades to all child resources)
3. Remove student namespaces (VMs, UDNs, quotas)
4. Clean up GitOps namespace

== Troubleshooting

=== Common Issues

==== Issue: Virtualization Operator Not Found

*Error*: `OpenShift Virtualization operator must be installed and ready`

*Solution*:
[source,bash]
----
# Install OpenShift Virtualization operator
oc create -f https://raw.githubusercontent.com/kubevirt/hyperconverged-cluster-operator/main/deploy/deploy.sh
----

==== Issue: Storage Class Not Available

*Error*: `No storage classes found on cluster`

*Solution*: Ensure at least one storage class exists. The role auto-selects RBD storage if available, otherwise uses the default storage class.

==== Issue: ArgoCD Application Not Syncing

*Symptom*: ArgoCD application stuck in "OutOfSync" state

*Solution*:
[source,bash]
----
# Force sync the parent application
oc patch application retail-edge-ha-parent \
  -n retail-edge-ha-gitops \
  --type merge \
  -p '{"operation":{"initiatedBy":{"automated":true}}}'
----

==== Issue: VirtualMachines Not Starting

*Symptom*: VMs created but not running

*Solution*: Check if `autoStart` is enabled in values or manually start VMs:
[source,bash]
----
virtctl start rhel-ha-node1 -n retail-edge-student-01
----
