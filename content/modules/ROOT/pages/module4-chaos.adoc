== Chaos Engineering Principles

Chaos engineering is the practice of deliberately introducing failures to:

* Validate system resilience
* Discover hidden dependencies
* Improve incident response
* Build confidence in production readiness

=== Real-World Retail Edge Failures

|**Network Partition**

== CAP Theorem and HA Trade-offs

Before testing chaos scenarios, understand the fundamental trade-offs each architecture makes.

=== CAP Theorem Overview

image::cap-theorem.png[CAP Theorem Triangle, align=center]

**The CAP Theorem states:** In a distributed system experiencing a network partition, you must choose between **Consistency** and **Availability** - you cannot have both.

=== How Each Architecture Handles CAP

[cols="1,1,1,2"]
|===
|Architecture |Chooses |Reasoning |During Network Partition

|**Pacemaker**
|**CP**
|Consistency + Partition Tolerance
|STONITH fences the isolated node. Surviving partition continues. Service is consistent but unavailable on fenced node.

|**MicroShift VRRP**
|**AP**
|Availability + Partition Tolerance
|VIP migrates to available node. Service remains available but both nodes might think they're active (split-brain risk without STONITH).

|**Two-Node OpenShift**
|**CP**
|Consistency + Partition Tolerance
|Without quorum (2/3 nodes), cluster becomes read-only. Prioritizes data consistency over availability.
|===

=== Hands-On CAP Demonstration

Let's prove the CAP theorem with a simple experiment.

**Scenario:** Isolate one Pacemaker node and observe the trade-off.

[source,bash,role=execute]
----
# Start Pacemaker VMs
virtctl start %RHEL_NODE1% -n %STUDENT_NAMESPACE%
virtctl start %RHEL_NODE2% -n %STUDENT_NAMESPACE%
----

Wait 2 minutes for cluster to form, then create a network partition:

[source,bash,role=execute]
----
virtctl ssh %VM_USER%@%RHEL_NODE1% -n %STUDENT_NAMESPACE% -- \
  sudo firewall-cmd --direct --add-rule ipv4 filter INPUT 0 -s 10.101.0.21 -j DROP
----

**Observe the CAP trade-off:**

1. **Consistency preserved**: Only one node has the VIP (no split-brain)
2. **Availability sacrificed**: Isolated node2 is fenced (becomes unavailable)
3. **Partition tolerance**: Node1 continues serving requests despite partition

Check cluster status:

[source,bash,role=execute]
----
virtctl ssh %VM_USER%@%RHEL_NODE1% -n %STUDENT_NAMESPACE% -- sudo pcs status
----

You'll see node2 offline - this is the **CP choice** in action.

Clean up:

[source,bash,role=execute]
----
virtctl ssh %VM_USER%@%RHEL_NODE1% -n %STUDENT_NAMESPACE% -- \
  sudo firewall-cmd --direct --remove-rule ipv4 filter INPUT 0 -s 10.101.0.21 -j DROP
----

**Key insight:** Pacemaker chose consistency (prevent split-brain) over availability (node2 fenced). This is the correct choice for stateful applications like databases.

=== CAP Trade-offs Summary

[cols="2,3,3"]
|===
|What You Want |Choose |Example Use Cases

|**Strong Consistency**
|CP (sacrifice availability during partition)
|Financial transactions, inventory counts, POS databases

|**High Availability**
|AP (accept eventual consistency)
|Content caching, session storage, read-heavy APIs

|**Both C and A**
|Not possible during network partition
|Only works in non-partitioned systems (single datacenter)
|===

In retail edge environments, **CP is usually preferred** because:

* Inventory must be accurate (can't sell what you don't have)
* Financial transactions must be consistent (no duplicate charges)
* Better to be temporarily unavailable than permanently inconsistent

== Scenario 1: Pacemaker Cluster - Network Partition

=== Test Setup

Start your Pacemaker VMs:

[source,bash,role=execute]
----
virtctl start %RHEL_NODE1% -n %STUDENT_NAMESPACE%
virtctl start %RHEL_NODE2% -n %STUDENT_NAMESPACE%
----

Wait for cluster to form:

[source,bash,role=execute]
----
sleep 120
----

Verify cluster is healthy:

[source,bash,role=execute]
----
virtctl ssh %VM_USER%@%RHEL_NODE1% -n %STUDENT_NAMESPACE% -- sudo pcs status
----

=== Chaos Experiment: Firewall-Induced Network Partition

SSH into node1:

[source,bash,role=execute]
----
virtctl ssh %VM_USER%@%RHEL_NODE1% -n %STUDENT_NAMESPACE%
----

**Hypothesis**: When node2 cannot communicate via Corosync, STONITH will fence it, and the VIP will remain on node1.

Block Corosync traffic from node2:

[source,bash]
----
sudo firewall-cmd --direct --add-rule ipv4 filter INPUT 0 -s 10.101.0.21 -p udp --dport 5405 -j DROP
----

Watch the cluster status:

[source,bash]
----
watch -n 2 sudo pcs status
----

Within 30-60 seconds, you should observe:

1. Node2 marked as `OFFLINE`
2. STONITH action: `fence-node2` executes
3. VirtualMachine `rhel-ha-node2` restarts
4. VIP remains on node1 (no unnecessary migration)

Press `Ctrl+C` to stop watching.

Remove the firewall rule:

[source,bash]
----
sudo firewall-cmd --direct --remove-rule ipv4 filter INPUT 0 -s 10.101.0.21 -p udp --dport 5405 -j DROP
----

Wait for node2 to rejoin:

[source,bash]
----
sleep 60
sudo pcs status
----

**Result**: STONITH fencing prevented split-brain. Node2 was forcibly rebooted and rejoined cleanly.

Exit:

[source,bash]
----
exit
----

=== Chaos Experiment: Simultaneous Process Crash

SSH into node1:

[source,bash,role=execute]
----
virtctl ssh %VM_USER%@%RHEL_NODE1% -n %STUDENT_NAMESPACE%
----

**Hypothesis**: Killing Corosync on both nodes simultaneously will trigger fencing, and the faster node will survive.

On node1, kill Corosync:

[source,bash]
----
sudo killall -9 corosync
----

Immediately SSH to node2 and kill Corosync there:

[source,bash,role=execute]
----
virtctl ssh %VM_USER%@%RHEL_NODE2% -n %STUDENT_NAMESPACE% -- sudo killall -9 corosync
----

Check which node gets fenced (check VM status from outside):

[source,bash,role=execute]
----
oc get vmi -n %STUDENT_NAMESPACE% -w
----

One VM should restart (fenced). Press `Ctrl+C` after observing.

**Result**: Pacemaker's fencing mechanism ensures only one node survives a simultaneous failure.

=== Chaos Experiment: CPU Saturation

SSH into Gateway A:

[source,bash,role=execute]
----
virtctl ssh %VM_USER%@%MICROSHIFT_GW_A% -n %STUDENT_NAMESPACE%
----

**Hypothesis**: Saturating CPU will cause health checks to fail, triggering VIP migration.

Create CPU load:

[source,bash]
----
stress-ng --cpu 4 --timeout 120s &
----

If `stress-ng` is not installed:

[source,bash]
----
for i in {1..4}; do yes > /dev/null & done
----

Monitor Keepalived logs in another window:

[source,bash]
----
sudo journalctl -u keepalived -f
----

Within 10-20 seconds, you should see:

[source,text]
----
VRRP_Script(check_microshift) failed
VRRP_Instance(MICROSHIFT_VIP) Entering BACKUP STATE
----

Check if VIP moved to Gateway B:

[source,bash,role=execute]
----
virtctl ssh %VM_USER%@%MICROSHIFT_GW_B% -n %STUDENT_NAMESPACE% -- ip addr show eth1 | grep %MICROSHIFT_VIP%
----

Stop the CPU load on Gateway A:

[source,bash]
----
killall yes
# or wait for stress-ng to finish
----

Wait for health check to recover:

[source,bash]
----
curl -k https://localhost:6443/readyz
----

VIP should migrate back to Gateway A within 30 seconds (higher priority).

Exit:

[source,bash]
----
exit
----

== Scenario 3: Two-Node OpenShift - etcd Quorum Loss

=== Test Setup

Start all two-node cluster VMs:

[source,bash,role=execute]
----
virtctl start %TWONODE_MASTER1% -n %STUDENT_NAMESPACE%
sleep 120
virtctl start %TWONODE_MASTER2% -n %STUDENT_NAMESPACE%
sleep 60
virtctl start %TWONODE_ARBITER% -n %STUDENT_NAMESPACE%
----

=== Chaos Experiment: Arbiter Disconnection

**Hypothesis**: Losing the arbiter will not affect cluster operations (master1 + master2 still have quorum).

Stop the arbiter:

[source,bash,role=execute]
----
virtctl stop %TWONODE_ARBITER% -n %STUDENT_NAMESPACE%
----

Verify connectivity between masters:

[source,bash,role=execute]
----
virtctl ssh core@%TWONODE_MASTER1% -n %STUDENT_NAMESPACE% -- ping -c 3 %TWONODE_IP2%
----

In a production cluster, you would observe:

[source,bash]
----
oc get etcd cluster -o jsonpath='{.status.conditions}' | jq
# Status: Available (2/3 quorum)
----

Restart the arbiter:

[source,bash,role=execute]
----
virtctl start %TWONODE_ARBITER% -n %STUDENT_NAMESPACE%
----

**Result**: Cluster continued operating with 2/3 quorum.

=== Chaos Experiment: Simultaneous Control-Plane Failure

**Hypothesis**: Losing both control-plane nodes will cause quorum loss and cluster unavailability.

Stop both masters:

[source,bash,role=execute]
----
virtctl stop %TWONODE_MASTER1% -n %STUDENT_NAMESPACE%
virtctl stop %TWONODE_MASTER2% -n %STUDENT_NAMESPACE%
----

Check that only the arbiter is running:

[source,bash,role=execute]
----
oc get vmi -n %STUDENT_NAMESPACE% | grep twonode
----

In a production cluster, this would result in:

* Only 1/3 etcd members online (arbiter)
* No quorum (need 2/3)
* API server unavailable
* All write operations fail

Restore quorum by starting master1:

[source,bash,role=execute]
----
virtctl start %TWONODE_MASTER1% -n %STUDENT_NAMESPACE%
----

Wait 2 minutes for startup, then start master2:

[source,bash,role=execute]
----
sleep 120
virtctl start %TWONODE_MASTER2% -n %STUDENT_NAMESPACE%
----

**Result**: Cluster becomes available again when quorum is restored.

=== Chaos Experiment: Network Latency Injection

SSH into master1:

[source,bash,role=execute]
----
virtctl ssh core@%TWONODE_MASTER1% -n %STUDENT_NAMESPACE%
----

**Hypothesis**: High latency to arbiter will cause etcd warnings but not immediate failure.

Add 200ms latency to arbiter traffic:

[source,bash]
----
sudo tc qdisc add dev eth1 root netem delay 200ms
----

In a production cluster, check etcd metrics:

[source,bash]
----
oc get --raw /metrics | grep etcd_network_peer_round_trip_time_seconds
# Should show increased latency
----

Remove the latency:

[source,bash]
----
sudo tc qdisc del dev eth1 root
----

Exit:

[source,bash]
----
exit
----

**Result**: etcd tolerates moderate latency but performance degrades. Keep arbiter <10ms RTT in production.

== Scenario 4: Combined Chaos - "Retail Nightmare"

This scenario simulates multiple simultaneous failures:

1. Power outage at edge location (both Pacemaker nodes down)
2. Network partition isolates MicroShift gateways
3. etcd arbiter unreachable

**Execute the chaos**:

[source,bash,role=execute]
----
# Pacemaker: Stop both nodes
virtctl stop %RHEL_NODE1% -n %STUDENT_NAMESPACE%
virtctl stop %RHEL_NODE2% -n %STUDENT_NAMESPACE%

# MicroShift: Stop Gateway A
virtctl stop %MICROSHIFT_GW_A% -n %STUDENT_NAMESPACE%

# Two-Node: Stop arbiter
virtctl stop %TWONODE_ARBITER% -n %STUDENT_NAMESPACE%
----

**Observe the impact**:

[source,bash,role=execute]
----
oc get vmi -n %STUDENT_NAMESPACE%
----

Count how many services are still available:

* Pacemaker VIP: ✗ (both nodes down)
* MicroShift VIP: ✓ (on Gateway B)
* Two-Node API: ✓ (master1 + master2 have quorum)

**Recovery**:

[source,bash,role=execute]
----
# Restore Pacemaker
virtctl start %RHEL_NODE1% -n %STUDENT_NAMESPACE%
virtctl start %RHEL_NODE2% -n %STUDENT_NAMESPACE%

# Restore MicroShift
virtctl start %MICROSHIFT_GW_A% -n %STUDENT_NAMESPACE%

# Restore arbiter
virtctl start %TWONODE_ARBITER% -n %STUDENT_NAMESPACE%
----

**Result**: 2 of 3 architectures survived simultaneous multi-component failures.

== Chaos Engineering Best Practices

1. **Start small**: Test single failures before combined scenarios
2. **Have a rollback plan**: Know how to restore service quickly
3. **Measure everything**: Metrics are evidence of resilience
4. **Run in production**: Staging doesn't reveal real dependencies

6. **Learn and improve**: Every failure is a learning opportunity

== Key Takeaways

* Pacemaker STONITH prevents split-brain scenarios
* VRRP health checks enable application-aware failover
* etcd quorum requires careful capacity planning

* Automatic recovery is faster than manual intervention
* Monitoring and alerting are critical for invisible failures

=== What is a GameDay?

A **GameDay** is a scheduled chaos engineering exercise where teams:

1. **Plan** failure scenarios in advance
2. **Execute** automated chaos injection
3. **Monitor** system behavior and recovery
4. **Document** findings and improvements
5. **Review** with stakeholders

=== Chaos Scenarios Injected

The script randomly selects from these failure types:

1. **VM Stop** (Hardware Failure)
   - Simulates server crash or power loss
   - Random VM is stopped via `virtctl stop`
   - Tests automatic recovery or manual restart

2. **Network Partition** (Split-Brain)
   - Simulates network cable unplug or switch failure
   - Firewall rule blocks communication between nodes
   - Tests STONITH fencing and quorum management

3. **CPU Saturation** (Resource Exhaustion)
   - Simulates runaway process or denial-of-service
   - Stress test runs for 30 seconds
   - Tests health check failures and failover

4. **Process Crash** (Software Bug)
   - Simulates application crash or segfault
   - Kills critical process (corosync, kube-apiserver)
   - Tests systemd automatic restart

=== Real-World GameDay Process

In production retail environments, GameDays follow this process:

**1 Week Before:**

* Announce GameDay to stakeholders (stores, NOC, management)
* Define failure scenarios and scope
* Create rollback procedures
* Schedule during low-traffic window (e.g., 2-5 AM)

**Day Of:**

* Pre-GameDay checklist: Verify backups, confirm on-call staffing
* Execute chaos script with telemetry enabled
* Monitor metrics dashboards (Grafana, RHACM)
* Document all observations and manual interventions

**Day After:**

* Post-GameDay report: Success rate, failures, lessons learned
* Action items: Fix issues discovered, improve runbooks
* Update disaster recovery procedures
* Share results with engineering and operations teams

**Monthly:**

* Repeat GameDays with increasing complexity
* Add new failure scenarios based on production incidents
* Measure improvement in recovery times
* Build organizational muscle memory

=== Safety Limits

The chaos script includes safety features:

* **Runs only in specified namespace** (isolated from other students)
* **Self-healing**: Most scenarios auto-recover within 60 seconds
* **Cleanup on exit**: Ctrl+C removes firewall rules and stops stress tests
* **Logs all actions**: Full audit trail in `/tmp/chaos-gameday-*.log`

**To stop early:**

Press `Ctrl+C` - the script will clean up and exit gracefully.

# Edit scripts/chaos-gameday.sh

# Add new scenario function
chaos_disk_full() {
    local vm="$1"
    log CHAOS "💾 Filling disk on ${vm}"

    # Fill /tmp to 90%
    virtctl ssh redhat@"${vm}" -n "${NAMESPACE}" -- \
        "dd if=/dev/zero of=/tmp/chaos-fill bs=1M count=1000 2>/dev/null || true"

    sleep 30

    # Clean up
    virtctl ssh redhat@"${vm}" -n "${NAMESPACE}" -- \
        "rm -f /tmp/chaos-fill"

    log INFO "✓ Disk pressure relieved on ${vm}"
}
