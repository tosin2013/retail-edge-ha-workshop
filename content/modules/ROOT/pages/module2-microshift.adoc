= Module 2: MicroShift with VRRP

== Overview

In this module, you'll deploy two MicroShift instances with Keepalived managing a Virtual IP using VRRP (Virtual Router Redundancy Protocol). This architecture provides high availability for edge gateways and load balancers.

NOTE: These VMs are **RHEL Device Edge** systems managed by **Red Hat Edge Manager** (flightctl). The devices enroll automatically with Edge Manager on boot, while the MicroShift Kubernetes layer can also register with ACM as a managed cluster. This dual-plane approach is how production edge gateways are managed at scale.

== Step 11: Understanding MicroShift Storage Limitations

This section explores a critical limitation of MicroShift: **local storage only**. Unlike Pacemaker with GFS2 (Module 1), MicroShift nodes cannot share storage. This has major implications for stateful applications.

=== Why This Matters in Retail

In retail edge environments, you might need:

* **Local databases** for POS transactions
* **Inventory caches** stored on disk
* **Configuration persistence** across reboots

With VRRP, the VIP migrates instantly, but **storage does NOT migrate** with it.

=== The Challenge

[cols="1,2,2"]
|===
|Architecture |Storage Behavior |What Happens on Failover

|**Pacemaker + GFS2**
|Shared storage (both nodes access same data)
|✅ VIP + storage migrate together

|**MicroShift + VRRP**
|Local storage (each node has its own disk)
|❌ VIP migrates, but data stays on failed node
|===

Let's demonstrate this limitation with a StatefulSet.

=== Deploy StatefulSet with Local Storage

Start Gateway A (if not already running):

[source,bash,role=execute,subs="attributes+"]
----
virtctl start {microshift-gw-a} -n {student-namespace}
----

Wait 60 seconds for MicroShift to be ready, then SSH in:

[source,bash,role=execute,subs="attributes+"]
----
virtctl ssh {vm-user}@{microshift-gw-a} -n {student-namespace}
----

Create a StatefulSet with PostgreSQL:

[source,role="execute"]
----
cat <<EOF | oc apply -f -
apiVersion: v1
kind: Service
metadata:
  name: postgres
spec:
  ports:
  - port: 5432
    name: postgres
  clusterIP: None
  selector:
    app: postgres
---
apiVersion: apps/v1
kind: StatefulSet
metadata:
  name: postgres
spec:
  serviceName: postgres
  replicas: 1
  selector:
    matchLabels:
      app: postgres
  template:
    metadata:
      labels:
        app: postgres
    spec:
      containers:
      - name: postgres
        image: postgres:15
        ports:
        - containerPort: 5432
          name: postgres
        env:
        - name: POSTGRES_PASSWORD
          value: redhat123
        - name: PGDATA
          value: /var/lib/postgresql/data/pgdata
        volumeMounts:
        - name: data
          mountPath: /var/lib/postgresql/data
  volumeClaimTemplates:
  - metadata:
      name: data
    spec:
      accessModes: [ "ReadWriteOnce" ]
      resources:
        requests:
          storage: 5Gi
EOF
----

Wait for the pod to start:

[source,role="execute"]

----

Press `Ctrl+C` when `postgres-0` shows `Running`.

=== Test Storage Behavior During Failover

Now, stop Gateway A to trigger VIP migration:

[source,role="execute"]
----
exit  # Exit SSH from Gateway A
----

[source,bash,role=execute,subs="attributes+"]
----
virtctl stop {microshift-gw-a} -n {student-namespace}
----

Wait 10 seconds for VRRP to migrate the VIP to Gateway B.

SSH into Gateway B:

[source,bash,role=execute,subs="attributes+"]
----
virtctl ssh {vm-user}@{microshift-gw-b} -n {student-namespace}
----

Check pod status:

[source,role="execute"]
----
oc get pods
----

You'll see one of these scenarios:

[source,text]
----
NAME         READY   STATUS    RESTARTS   AGE
postgres-0   0/1     Pending   0          5m
----

OR

[source,text]
----
NAME         READY   STATUS             RESTARTS   AGE
postgres-0   0/1     CrashLoopBackOff   3          5m
----

Why is the pod not running on Gateway B?

Check the PersistentVolumeClaim:

[source,role="execute"]
----
oc get pvc
----

[source,text]
----
NAME             STATUS   VOLUME   CAPACITY   ACCESS MODES   STORAGECLASS
data-postgres-0  Bound    pvc-...  5Gi        RWO            ...
----

The PVC is bound to a volume on **Gateway A's local storage**! Gateway B cannot access it because MicroShift uses `ReadWriteOnce` (RWO) - only one node can mount at a time.

Try to access the database via the VIP:

[source,role="execute",subs="attributes+"]
----
curl -v telnet://{microshift-vip}:5432 --max-time 5
----

This will timeout - the database pod cannot start on Gateway B.

=== The Problem

[source,text]
----
┌─────────────────┐       ┌─────────────────┐
│ Gateway A       │       │ Gateway B       │
│ (STOPPED)       │       │ (VIP: MASTER)   │
├─────────────────┤       ├─────────────────┤
│ Pod: postgres-0 │       │ Pod: Pending    │
│ PVC: Bound      │ ─X──► │ PVC: Inaccessible
│ Data: 3 rows    │       │ Data: None      │
└─────────────────┘       └─────────────────┘
      │                           │
      │ Local Storage (RWO)       │ Cannot mount
      └─────────────────────────  │
                                  ✗
----

**Key insight:** VRRP migrated the VIP (network access) but the StatefulSet pod is stuck because its data is on Gateway A!

=== Solution: Application-Level Replication

For stateful applications on MicroShift, you must use application-level replication:

[cols="1,2,2"]
|===
|Database |Replication Method |How It Works

|**PostgreSQL**
|Streaming Replication
|Primary → standby, promote standby on failover

|**MySQL**
|InnoDB Cluster (Group Replication)
|Multi-primary, automatic failover

|**MongoDB**
|Replica Set
|3+ replicas, automatic primary election
|===

=== Real-World Retail Architecture

image::module2-storage-architecture.png[MicroShift Storage Architecture, align=center,link=self,window=blank]

In production, you would deploy:

1. **PostgreSQL on both gateways** with streaming replication
2. **VIP points to current primary** (same as Module 1 optional exercise)
3. **On failure:** VRRP migrates VIP, PostgreSQL on backup promotes to primary
4. **Data survives** because it's replicated, not shared

=== Key Takeaways

* ❌ MicroShift does NOT support shared storage (no ReadWriteMany)
* ✅ VRRP migrates the VIP only, not data
* ✅ StatefulSets remain bound to the node where data exists
* ✅ Applications must handle data replication themselves
* 💡 MicroShift is best for:
  ** Stateless workloads (web apps, proxies, load balancers)
  ** Stateful apps with built-in replication (databases, message queues)
  ** Edge gateways where network HA is more important than storage HA

=== Comparison: Pacemaker vs. MicroShift Storage

[cols="1,2,2"]
|===
|Feature |Pacemaker (Module 1) |MicroShift (Module 2)

|**Shared Storage**
|✅ Yes (GFS2, NFS, SAN)
|❌ No (local storage only)

|**Failover Scope**
|VIP + storage + application
|VIP only

|**Data Migration**
|Automatic (same PVC on both nodes)
|Not possible (PVC stays on original node)

|**Recovery Time**
|30-60 seconds (full stack)
|3-5 seconds (VIP only, app may not start)

|**Complexity**
|Higher (cluster + fencing + GFS2)
|Lower (just VRRP)

|**Best For**
|Stateful HA (databases, file servers)
|Stateless HA (gateways, proxies)
|===

== MicroShift Fleet Management

Now that you've configured MicroShift HA, consider how this scales in production:

=== Edge Manager + MicroShift

Each MicroShift gateway is managed on **two levels**:

[cols="1,2,2"]
|===
|Layer |Managed By |What It Does

|**Device (RHEL OS)**
|Red Hat Edge Manager (flightctl)
|OS image updates, device configuration, health monitoring

|**Kubernetes (MicroShift)**
|MicroShift local control plane
|Pod scheduling, service networking, container lifecycle
|===

Verify your devices are still healthy in Edge Manager and review the fleet:

[source,bash,role=execute]
----
flightctl get devices
----

[source,bash,role=execute]
----
flightctl get fleets
----

Throughout this module, all the device-level configuration -- hostname, keepalived, firewall rules, MicroShift enablement -- was delivered by Fleet Manager from the workshop Git repository. The `microshift-vrrp` fleet ensures every gateway gets the same configuration automatically.

In production, Edge Manager fleets allow you to:

* **Group gateways by region** -- push VRRP configuration updates to all West Coast stores at once
* **Roll out MicroShift updates** -- upgrade the edge Kubernetes runtime across thousands of devices
* **Monitor device health** -- detect offline gateways before store staff notice
* **Enforce compliance** -- ensure all gateways run the approved OS image and configuration
* **Git-driven configuration** -- all device configs live in Git; changes are automatically pushed to matching fleets

== Key Takeaways

* VRRP provides lightweight, protocol-level HA for IP addresses
* Keepalived monitors application health (MicroShift API)
* Priority determines MASTER selection (100 > 90)
* Failover is automatic and typically completes in 3-5 seconds
* Preemption returns VIP to higher-priority node when it recovers
* Layer 2 multicast (224.0.0.18) is required for VRRP
* Red Hat Edge Manager provides fleet-level management for the RHEL device layer
