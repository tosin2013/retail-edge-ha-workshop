= Module 2: MicroShift with VRRP

== Overview

In this module, you'll deploy two MicroShift instances with Keepalived managing a Virtual IP using VRRP (Virtual Router Redundancy Protocol). This architecture provides high availability for edge gateways and load balancers.

[source,text]
----
┌─────────────────────┐      ┌─────────────────────┐

│  10.102.0.20        │      │  10.102.0.21        │
│  Priority: 100      │◄────►│  Priority: 90       │
│  (MASTER)           │ VRRP │  (BACKUP)           │
└─────────────────────┘      └─────────────────────┘
         │                            │
         └────────────────────────────┘
           Virtual IP: 10.102.0.100
         (floats to highest priority)
----

*Key Features:*

* **MicroShift**: Lightweight Kubernetes for edge
* **VRRP**: Multicast advertisements (224.0.0.18)
* **Health Check**: Monitors MicroShift API (port 6443)
* **Automatic Failover**: 3-5 second VIP migration

By the end of this module, you will:

== Step 1: Verify and Start VMs

Check your MicroShift VMs:

[source,bash,role=execute]
----
oc get vm -n %STUDENT_NAMESPACE% | grep microshift
----

Start both gateways:

[source,bash,role=execute]
----
virtctl start %MICROSHIFT_GW_A% -n %STUDENT_NAMESPACE%
----

[source,bash,role=execute]
----
virtctl start %MICROSHIFT_GW_B% -n %STUDENT_NAMESPACE%
----

Wait for VMs to be ready (about 3-4 minutes for MicroShift to start):

[source,bash,role=execute]
----
oc get vmi -n %STUDENT_NAMESPACE% -w
----

Press `Ctrl+C` when both show `Running`.

== Step 2: Verify MicroShift Installation

SSH into Gateway A:

[source,bash,role=execute]
----
virtctl ssh %VM_USER%@%MICROSHIFT_GW_A% -n %STUDENT_NAMESPACE%
----

Password: `%VM_PASSWORD%`

Check MicroShift status:

[source,bash]
----
sudo systemctl status microshift
----

You should see `active (running)`.

Verify MicroShift is ready:

[source,bash]
----
curl -k https://localhost:6443/readyz
----

Expected output: `ok`

Check the kubeconfig:

[source,bash]
----
oc get nodes
----

You should see the MicroShift node in `Ready` state.

Check network configuration:

[source,bash]
----
ip addr show eth1
----

You should see:

[source,text]
----
inet 10.102.0.20/24 brd 10.102.0.255 scope global eth1
----

Exit the session:

[source,bash]
----
exit
----

== Step 3: Verify Keepalived Configuration

SSH back into Gateway A:

[source,bash,role=execute]
----
virtctl ssh %VM_USER%@%MICROSHIFT_GW_A% -n %STUDENT_NAMESPACE%
----

Check Keepalived status:

[source,bash]
----
sudo systemctl status keepalived
----

View the Keepalived configuration:

[source,bash]
----
sudo cat /etc/keepalived/keepalived.conf
----

Note the configuration:

* `state MASTER` - Gateway A is the primary
* `priority 100` - Higher than Gateway B (90)
* `virtual_router_id 1` - Shared VRID
* `virtual_ipaddress { 10.102.0.100/24 }` - The VIP

Check VRRP advertisements in the logs:

[source,bash]
----
sudo journalctl -u keepalived --no-pager -n 20 | grep VRRP
----

You should see entries like:

[source,text]
----
VRRP_Instance(MICROSHIFT_VIP) Entering MASTER STATE
----

== Step 4: Verify Virtual IP

Check if the VIP is assigned to Gateway A:

[source,bash]
----
ip addr show eth1 | grep %MICROSHIFT_VIP%
----

You should see:

[source,text]
----
inet 10.102.0.100/24 scope global secondary eth1
----

The `secondary` keyword indicates this is the virtual IP.

Test VIP connectivity:

[source,bash]
----
ping -c 3 %MICROSHIFT_VIP%
----

Access MicroShift API via the VIP:

[source,bash]
----
curl -k https://%MICROSHIFT_VIP%:6443/readyz
----

Expected: `ok`

Exit the session:

[source,bash]
----
exit
----

== Step 6: Test Graceful Failover

Stop Gateway A to trigger VIP migration:

[source,bash,role=execute]
----
virtctl stop %MICROSHIFT_GW_A% -n %STUDENT_NAMESPACE%
----

Wait 10 seconds for VRRP failover, then SSH into Gateway B:

[source,bash,role=execute]
----
virtctl ssh %VM_USER%@%MICROSHIFT_GW_B% -n %STUDENT_NAMESPACE%
----

Check if the VIP has migrated:

[source,bash]
----
ip addr show eth1 | grep %MICROSHIFT_VIP%
----

You should now see:

[source,text]
----
inet 10.102.0.100/24 scope global secondary eth1
----

Check the Keepalived logs:

[source,bash]
----
sudo journalctl -u keepalived --no-pager -n 30 | grep VRRP
----

Look for:

[source,text]
----
VRRP_Instance(MICROSHIFT_VIP) Transition to MASTER STATE
----

Verify the VIP responds:

[source,bash]
----
ping -c 3 %MICROSHIFT_VIP%
curl -k https://%MICROSHIFT_VIP%:6443/readyz
----

Exit the session:

[source,bash]
----
exit
----

Start Gateway A again:

[source,bash,role=execute]

Wait 60 seconds for MicroShift and Keepalived to start, then SSH in:

[source,bash,role=execute]

----

The VIP should be back on Gateway A because it has higher priority (100 vs 90).

Check Keepalived logs:

[source,bash]
----
sudo journalctl -u keepalived --no-pager -n 30 | grep VRRP
----

Exit:

[source,bash]
----

----

== Step 8: Deploy Test Workload

SSH into the active gateway (check which has the VIP):

[source,bash,role=execute]
----
virtctl ssh %VM_USER%@%MICROSHIFT_GW_A% -n %STUDENT_NAMESPACE%
----

Create a test deployment:

[source,bash]
----
oc create deployment nginx --image=nginx --replicas=2
----

Expose the deployment:

[source,bash]
----
oc expose deployment nginx --port=80 --type=NodePort
----

Get the NodePort:

[source,bash]
----
oc get svc nginx
----

Note the port (e.g., `80:31234/TCP`).

Test access via the VIP:

[source,bash]
----
curl http://%MICROSHIFT_VIP%:<nodeport>
----

You should see the nginx welcome page.

== Step 9: Test Workload During Failover

Open a second terminal and run continuous requests:

[source,bash]
----
while true; do
  curl -s -o /dev/null -w "Status: %{http_code}\n" http://%MICROSHIFT_VIP%:<nodeport>
  sleep 1
done
----

In the first terminal, stop Gateway A:

[source,bash,role=execute]
----
virtctl stop %MICROSHIFT_GW_A% -n %STUDENT_NAMESPACE%
----

Watch the second terminal. You should see:

* 1-2 failed requests during VIP migration (3-5 seconds)
* Automatic recovery when VIP moves to Gateway B
* Continued service availability

Press `Ctrl+C` to stop the loop.

== Step 10: Validate Health Check

SSH into Gateway B (now MASTER):

[source,bash,role=execute]
----
virtctl ssh %VM_USER%@%MICROSHIFT_GW_B% -n %STUDENT_NAMESPACE%
----

Stop MicroShift (simulating API failure):

[source,bash]
----
sudo systemctl stop microshift
----

Watch Keepalived detect the failure:

[source,bash]
----
sudo journalctl -u keepalived -f
----

Within 10 seconds, you should see:

[source,text]
----
VRRP_Script(check_microshift) failed
VRRP_Instance(MICROSHIFT_VIP) Received lower priority advert
----

However, since Gateway A is still stopped, Gateway B will retain MASTER state even with failed health check (no better alternative).

Restart MicroShift:

[source,bash]
----
sudo systemctl start microshift
----

Wait for MicroShift to be ready:

[source,bash]
----
watch -n 2 curl -k -s https://localhost:6443/readyz
----

Press `Ctrl+C` when it returns `ok`.

Exit:

[source,bash]
----
exit
----

== Troubleshooting

[%collapsible]
====
*Problem:* VIP not appearing on either node

*Solution:*
[source,bash]
----
sudo systemctl restart keepalived
sudo journalctl -u keepalived --no-pager -n 50
----

*Problem:* VRRP advertisements not seen

*Solution:* Check firewall:
[source,bash]
----
sudo firewall-cmd --list-all
# Should show: protocols: vrrp
----

*Problem:* MicroShift not starting

*Solution:*
[source,bash]
----
sudo journalctl -u microshift --no-pager -n 50
sudo systemctl restart microshift
----
====

== Key Takeaways

* VRRP provides lightweight, protocol-level HA for IP addresses
* Keepalived monitors application health (MicroShift API)
* Priority determines MASTER selection (100 > 90)
* Failover is automatic and typically completes in 3-5 seconds
* Preemption returns VIP to higher-priority node when it recovers
* Layer 2 multicast (224.0.0.18) is required for VRRP
