= Module 1: Pacemaker HA Cluster

== Overview

In this module, you'll deploy a two-node Pacemaker/Corosync cluster with STONITH (Shoot The Other Node In The Head) fencing. This architecture is ideal for stateful applications that require quick failover.

== Architecture Comparison

=== Lab Environment (What You'll Build)

image::module1-lab-architecture.png[Lab Architecture - VMs with Pacemaker, align=center,link=self,window=blank]

=== Real-World Deployment (Retail Store)

image::module1-real-architecture.png[Real-World Architecture - Physical Servers, align=center,link=self,window=blank]

=== Key Differences

[cols="1,2,2"]
|===
|Component |Lab Environment |Real-World Deployment

|**Infrastructure**
|Virtual Machines on OpenShift/KubeVirt
|Physical servers (Dell, HP, Lenovo)

|**Fencing Method**
|`fence_kubevirt` (VM API calls)
|`fence_ipmilan` (IPMI/iDRAC), `fence_apc` (PDU), `fence_vmware`

|**Network**
|User Defined Network (UDN) - Layer 2
|Dedicated heartbeat NIC + VLAN

|**Storage**
|Ceph RBD (OCS/ODF)
|Local SSD RAID or SAN (iSCSI/FC)

|**Use Case**
|Learning HA concepts
|Store POS system, local database, inventory

|**Recovery Time**
|~30 seconds (VM restart)
|~60-90 seconds (physical server boot)

|**Cost**
|Zero (shared OpenShift cluster)
|$10-15K per store (2 servers + network)
|===

*What's the Same:*

* Pacemaker/Corosync clustering software
* STONITH fencing requirement
* Virtual IP for application connectivity
* Active/Passive failover pattern
* Configuration commands (`pcs`, `crm`)

== Step 4: Configure Pacemaker Cluster

SSH into node1:

[source,bash,role=execute,subs="attributes+"]
----
virtctl ssh {vm-user}@{rhel-node1} -n {student-namespace}
----

Authenticate the cluster nodes (password: `redhat`):

[source,role="execute"]
----
sudo pcs host auth rhel-ha-node1 rhel-ha-node2 -u hacluster -p redhat
----

Create the cluster:

[source,role="execute"]
----
sudo pcs cluster setup ha-cluster rhel-ha-node1 rhel-ha-node2 \
  --transport udp \
  --force
----

Start the cluster:

[source,role="execute"]
----
sudo pcs cluster start --all
----

Enable cluster auto-start:

[source,role="execute"]
----
sudo pcs cluster enable --all
----

== Step 5: Configure Cluster Properties

Disable STONITH temporarily (we'll enable it after configuring fencing):

[source,role="execute"]
----
sudo pcs property set stonith-enabled=false
----

Set quorum policy for two-node cluster:

[source,role="execute"]
----
sudo pcs property set no-quorum-policy=ignore
----

Check cluster status:

[source,role="execute"]
----
sudo pcs status
----

You should see both nodes online.

== Step 6: Configure STONITH Fencing

Create a ServiceAccount for fence_kubevirt:

[source,role="execute"]
----
exit
----

Back in the terminal, create the ServiceAccount:

[source,bash,role=execute,subs="attributes+"]
----
oc create sa pacemaker-fencing -n {student-namespace}
----

Grant permissions to manage VMs:

[source,bash,role=execute,subs="attributes+"]
----
oc create role vm-manager \
  --verb=get,list,update,patch,delete \
  --resource=virtualmachines,virtualmachineinstances \
  -n {student-namespace}
----

[source,bash,role=execute,subs="attributes+"]
----
oc create rolebinding vm-manager-binding \
  --role=vm-manager \
  --serviceaccount={student-namespace}:pacemaker-fencing \
  -n {student-namespace}
----

Get the ServiceAccount token:

[source,bash,role=execute,subs="attributes+"]
----
oc create token pacemaker-fencing -n {student-namespace} --duration=8760h
----

*Copy the token output* - you'll need it in the next step.

== Step 7: Configure fence_kubevirt

SSH back into node1:

[source,bash,role=execute,subs="attributes+"]
----
virtctl ssh {vm-user}@{rhel-node1} -n {student-namespace}
----

Create fence device for node1:

[source,role="execute",subs="attributes+"]
----
sudo pcs stonith create fence-node1 fence_kubevirt \
  token="<paste-token-here>" \
  plug=rhel-ha-node1 \
  namespace={student-namespace} \
  apiserveraddr={cluster-api} \
  pcmk_host_list="rhel-ha-node1" \
  pcmk_reboot_action="reboot" \
  ssl_insecure=1
----

Create fence device for node2:

[source,role="execute",subs="attributes+"]
----
sudo pcs stonith create fence-node2 fence_kubevirt \
  token="<paste-token-here>" \
  plug=rhel-ha-node2 \
  namespace={student-namespace} \
  apiserveraddr={cluster-api} \
  pcmk_host_list="rhel-ha-node2" \
  pcmk_reboot_action="reboot" \
  ssl_insecure=1
----

Add location constraints (fence devices on opposite nodes):

[source,role="execute"]
----
sudo pcs constraint location fence-node1 avoids rhel-ha-node1
sudo pcs constraint location fence-node2 avoids rhel-ha-node2
----

Enable STONITH:

[source,role="execute"]
----
sudo pcs property set stonith-enabled=true
----

Test fencing (this will reboot node2):

[source,role="execute"]
----
sudo pcs stonith fence rhel-ha-node2
----

Wait 30 seconds for node2 to reboot and rejoin:

[source,role="execute"]
----
watch -n 2 sudo pcs status
----

Press `Ctrl+C` when node2 shows online again.

== Step 11: Shared Storage with Pacemaker

In retail environments, shared storage enables critical capabilities like shared POS transaction logs, replicated inventory databases, and synchronized configuration files. Let's configure a clustered filesystem that automatically fails over with the cluster.

image::module1-storage-architecture.png[Pacemaker Shared Storage Architecture, align=center,link=self,window=blank]

=== Why Shared Storage Matters

*Retail use cases:*

* **POS Transaction Logs**: All sales data available on both nodes for immediate failover
* **Inventory Database**: Real-time stock levels accessible from either server
* **Configuration Files**: Consistent settings across the HA pair
* **Store Reports**: Generated on one node, accessible from both

=== Create Shared PersistentVolumeClaim

First, exit any SSH session and create a shared PVC from your terminal:

[source,bash,role=execute,subs="attributes+"]
----
cat <<EOF | oc apply -f -
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: pacemaker-shared-storage
  namespace: {student-namespace}
spec:
  accessModes:
    - ReadWriteMany
  resources:
    requests:
      storage: 10Gi
  storageClassName: ocs-external-storagecluster-cephfs
EOF
----

Verify the PVC is bound:

[source,bash,role=execute,subs="attributes+"]
----
oc get pvc pacemaker-shared-storage -n {student-namespace}
----

You should see `STATUS: Bound`.

NOTE: ReadWriteMany (RWX) access mode allows both VMs to mount the volume simultaneously - essential for clustered filesystems.

=== Attach Shared Storage to VMs

Stop both VMs to attach the storage:

[source,bash,role=execute,subs="attributes+"]
----
virtctl stop {rhel-node1} -n {student-namespace}
virtctl stop {rhel-node2} -n {student-namespace}
----

Wait for VMs to stop (check with `oc get vmi -n {student-namespace}`).

Patch node1 to add the shared disk:

[source,bash,role=execute,subs="attributes+"]
----
oc patch vm rhel-ha-node1 -n {student-namespace} --type=merge -p '
spec:
  template:
    spec:
      domain:
        devices:
          disks:
          - name: containerdisk
            disk:
              bus: virtio
          - name: cloudinitdisk
            disk:
              bus: virtio
          - name: shared-data
            disk:
              bus: virtio
      volumes:
      - name: containerdisk
        containerDisk:
          image: registry.redhat.io/rhel9/rhel-guest-image:latest
      - name: cloudinitdisk
        cloudInitNoCloud:
          userData: |
            # (existing cloud-init - unchanged)
      - name: shared-data
        persistentVolumeClaim:
          claimName: pacemaker-shared-storage
'
----

Repeat for node2:

[source,bash,role=execute,subs="attributes+"]
----
oc patch vm rhel-ha-node2 -n {student-namespace} --type=merge -p '
spec:
  template:
    spec:
      domain:
        devices:
          disks:
          - name: containerdisk
            disk:
              bus: virtio
          - name: cloudinitdisk
            disk:
              bus: virtio
          - name: shared-data
            disk:
              bus: virtio
      volumes:
      - name: containerdisk
        containerDisk:
          image: registry.redhat.io/rhel9/rhel-guest-image:latest
      - name: cloudinitdisk
        cloudInitNoCloud:
          userData: |
            # (existing cloud-init - unchanged)
      - name: shared-data
        persistentVolumeClaim:
          claimName: pacemaker-shared-storage
'
----

IMPORTANT: In production, you would use DataVolumes or edit VM manifests directly. This patch command is simplified for workshop purposes.

Start the VMs:

[source,bash,role=execute,subs="attributes+"]
----
virtctl start {rhel-node1} -n {student-namespace}
virtctl start {rhel-node2} -n {student-namespace}
----

Wait for VMs to boot (~30 seconds).

=== Configure GFS2 Clustered Filesystem

SSH into node1:

[source,bash,role=execute,subs="attributes+"]
----
virtctl ssh {vm-user}@{rhel-node1} -n {student-namespace}
----

Check that the shared disk appears as `/dev/vdb`:

[source,role="execute"]
----
lsblk
----

You should see:

[source,text]
----
NAME   MAJ:MIN RM SIZE RO TYPE MOUNTPOINT
vda    252:0    0  30G  0 disk
└─vda1 252:1    0  30G  0 part /
vdb    252:16   0  10G  0 disk
----

Install GFS2 utilities (pre-installed via cloud-init):

[source,role="execute"]
----
sudo dnf install -y gfs2-utils dlm
----

Create GFS2 filesystem on the shared disk:

[source,role="execute"]
----
sudo mkfs.gfs2 -p lock_dlm -t ha-cluster:shared-data -j 2 /dev/vdb
----

Parameters explained:
* `-p lock_dlm`: Use DLM (Distributed Lock Manager) for cluster locking
* `-t ha-cluster:shared-data`: Cluster name and filesystem name
* `-j 2`: Two journals (one per node)

Create mount point:

[source,role="execute"]
----
sudo mkdir -p /mnt/shared
----

=== Add Filesystem Resource to Pacemaker

Create a Pacemaker resource for the GFS2 filesystem:

[source,role="execute"]
----
sudo pcs resource create shared-fs Filesystem \
  device="/dev/vdb" \
  directory="/mnt/shared" \
  fstype="gfs2" \
  options="noatime" \
  op monitor interval=20s \
  clone interleave=true
----

The `clone` parameter allows both nodes to mount the filesystem simultaneously.

Check the resource status:

[source,role="execute"]
----
sudo pcs status resources
----

You should see `shared-fs-clone` started on both nodes.

Verify the filesystem is mounted:

[source,role="execute"]
----
mount | grep /mnt/shared
----

Expected output:

[source,text]
----
/dev/vdb on /mnt/shared type gfs2 (rw,noatime)
----

=== Real-World Storage Architecture

In production retail deployments:

[cols="1,2,2"]
|===
|Storage Type |Technology |Use Case

|**Shared SAN**
|iSCSI, Fibre Channel
|High-performance database storage

|**Clustered Filesystem**
|GFS2 (RHEL), OCFS2
|Configuration files, logs, reports

|**Network Storage**
|NFS, Ceph RBD
|Backups, large file shares

|**Local RAID**
|Hardware RAID 1/10
|OS disks, temporary data
|===

*Key difference from this lab:*

* Lab uses Ceph RBD (ODF) over network
* Production stores typically use local SAN (lower latency, higher cost)
* Both support ReadWriteMany for GFS2

=== Key Takeaways

* ✅ GFS2 enables concurrent filesystem access from multiple nodes
* ✅ Pacemaker manages mounting/unmounting automatically
* ✅ Fencing prevents data corruption during split-brain scenarios
* ✅ Shared storage requires ReadWriteMany (RWX) access mode
* ✅ Real retail stores use GFS2 for transaction logs and shared data
* ⚠️ Performance depends on storage backend (local SAN is fastest)

== Optional: Advanced Exercise - Database HA with Fleet Manager

This optional exercise demonstrates a real-world scenario: deploying highly-available PostgreSQL databases across multiple retail stores using Red Hat Advanced Cluster Management (ACM) Fleet Manager.

=== Scenario: Multi-Store POS Database Deployment

image::module1-fleet-manager.png[Fleet Manager Multi-Store Deployment, align=center,link=self,window=blank]

=== Business Value

**Why this matters in retail:**

* **Zero downtime during failures**: Store operations continue even if a database server dies
* **Local data**: Each store has its own database (works during WAN outages)
* **Centralized management**: HQ deploys updates to 1000s of stores from one location
* **Consistent configuration**: All stores use identical HA setup (no configuration drift)

=== Exercise Overview

You'll configure PostgreSQL with streaming replication and add it as a Pacemaker resource. This creates an active/standby database cluster where:

1. **Node 1**: Runs PostgreSQL as primary (accepts writes)
2. **Node 2**: Runs PostgreSQL as standby (receives replication stream)
3. **VIP**: Applications always connect to VIP, which follows the primary
4. **On failure**: Pacemaker promotes standby → primary, moves VIP, app reconnects automatically

=== Step 1: Install PostgreSQL

SSH into node1:

[source,bash,role=execute,subs="attributes+"]
----
virtctl ssh {vm-user}@{rhel-node1} -n {student-namespace}
----

Install PostgreSQL on both nodes:

[source,role="execute"]
----
# On node1
sudo dnf install -y postgresql-server postgresql-contrib

# SSH to node2 and install there too
ssh rhel-ha-node2
sudo dnf install -y postgresql-server postgresql-contrib
exit
----

=== Step 2: Configure Primary Database (Node 1)

Initialize PostgreSQL:

[source,role="execute"]
----
sudo -u postgres /usr/bin/postgresql-setup --initdb
----

Edit PostgreSQL configuration for replication:

[source,role="execute"]
----
sudo tee -a /var/lib/pgsql/data/postgresql.conf > /dev/null <<EOF
listen_addresses = '*'
wal_level = replica
max_wal_senders = 3
max_replication_slots = 3
hot_standby = on
EOF
----

Configure authentication for replication:

[source,role="execute"]
----
sudo tee -a /var/lib/pgsql/data/pg_hba.conf > /dev/null <<EOF
host replication replicator 10.101.0.21/32 md5
host all all 10.101.0.0/24 md5
EOF
----

Start PostgreSQL and create replication user:

[source,role="execute"]
----
sudo systemctl start postgresql
sudo systemctl enable postgresql

sudo -u postgres psql <<EOF
CREATE USER replicator REPLICATION LOGIN ENCRYPTED PASSWORD 'redhat';
CREATE DATABASE storedb;
EOF
----

=== Step 3: Configure Standby Database (Node 2)

Stop PostgreSQL on node2 and clone from primary:

[source,role="execute"]
----
ssh rhel-ha-node2

sudo systemctl stop postgresql
sudo -u postgres rm -rf /var/lib/pgsql/data/*

sudo -u postgres pg_basebackup -h 10.101.0.20 -U replicator -D /var/lib/pgsql/data -Fp -Xs -R -P
----

When prompted for password, enter: `redhat`

Create standby signal file:

[source,role="execute"]
----
sudo -u postgres touch /var/lib/pgsql/data/standby.signal
----

Start PostgreSQL on standby:

[source,role="execute"]
----
sudo systemctl start postgresql
sudo systemctl enable postgresql
exit
----

=== Step 4: Verify Replication

On node1, check replication status:

[source,role="execute"]
----
sudo -u postgres psql -c "SELECT * FROM pg_stat_replication;"
----

You should see node2 connected as a replication client.

Test replication:

[source,role="execute"]
----
# Create test table on primary (node1)
sudo -u postgres psql -d storedb <<EOF
CREATE TABLE test_replication (
    id SERIAL PRIMARY KEY,
    message TEXT,
    created_at TIMESTAMP DEFAULT NOW()
);

INSERT INTO test_replication (message) VALUES ('Replication test from primary');
EOF
----

On node2, verify the data replicated:

[source,role="execute"]
----
ssh rhel-ha-node2
sudo -u postgres psql -d storedb -c "SELECT * FROM test_replication;"
exit
----

You should see the test row (replication working!).

=== Step 5: Add PostgreSQL to Pacemaker

Stop PostgreSQL on both nodes (Pacemaker will manage it):

[source,role="execute"]
----
sudo systemctl stop postgresql
sudo systemctl disable postgresql

ssh rhel-ha-node2 'sudo systemctl stop postgresql && sudo systemctl disable postgresql'
----

Create Pacemaker resource for PostgreSQL:

[source,role="execute"]
----
sudo pcs resource create postgresql-db systemd:postgresql \
  op start timeout=60s \
  op stop timeout=60s \
  op monitor interval=10s timeout=30s
----

Create resource group (VIP + Database together):

[source,role="execute"]
----
sudo pcs resource group add postgres-ha cluster-vip postgresql-db
----

This ensures the VIP and database always run on the same node.

=== Step 6: Test Database Failover

Verify the resource group is running:

[source,role="execute"]
----
sudo pcs status
----

You should see `postgres-ha` group with both `cluster-vip` and `postgresql-db` on the same node.

Connect to database via VIP:

sudo -u postgres psql -h {pacemaker-vip} -d storedb -c "SELECT * FROM test_replication;"

Trigger failover by putting the active node into standby:

----

Watch the migration:

----

The `postgres-ha` group should migrate to node2, including:
1. VIP moves to node2
2. PostgreSQL stops on node1
3. PostgreSQL starts on node2
4. Standby is promoted to primary

Connect to database again (same VIP, now on node2):

[source,role="execute",subs="attributes+"]
----

Application sees no difference - same VIP, same connection string!

Bring node1 back online:

[source,role="execute"]

----

=== Step 8: Test During Failure

Simulate a transaction happening during failover:

Open a new terminal and run this loop (simulates POS system):

[source,role="execute",subs="attributes+"]
----
while true; do
  sudo -u postgres psql -h {pacemaker-vip} -d storedb -c \
    "INSERT INTO transactions (sku, quantity, total) VALUES ('PROD-002', 1, 5.49); SELECT NOW() as inserted_at;"
  sleep 2
done
----

While that's running, in your main terminal trigger failover:

[source,role="execute"]
----
ssh rhel-ha-node2
sudo systemctl stop corosync  # Simulate node2 crash
exit
----

Watch the cluster:

[source,role="execute"]
----
watch -n 2 sudo pcs status
----

You'll see:
1. Node2 goes OFFLINE (lost heartbeat)
2. STONITH fences node2 (VM reboots)
3. Transaction loop may error for 10-30 seconds
4. Standby promoted to primary
5. Transactions resume automatically

After node2 rejoins, check transaction history:

[source,role="execute",subs="attributes+"]
----
sudo -u postgres psql -h {pacemaker-vip} -d storedb -c \
  "SELECT COUNT(*) as total_transactions, MAX(timestamp) as last_transaction FROM transactions;"
----

Some transactions may have been lost during the 10-30 second outage, but the store is back online!

=== Real-World Considerations

image::module1-failover-flow.png[Database Failover Flow with STONITH, align=center,link=self,window=blank]

**Why STONITH is critical for databases:**

* **Prevents split-brain**: Without fencing, both nodes might think they're primary
* **Data integrity**: Only one node can accept writes at a time
* **Automatic recovery**: No manual intervention needed during failures

**Typical retail database HA metrics:**

* **RPO (Recovery Point Objective)**: 0 seconds (synchronous replication)
* **RTO (Recovery Time Objective)**: 30-60 seconds (failover + promotion)
* **Availability**: 99.95% (4.4 hours downtime per year)

=== Fleet Manager Integration (Conceptual)

In a real deployment with ACM, you would:

1. **Create ManagedClusterSet** for retail stores
2. **Define Placement Policy** (which stores get database HA)
3. **Create GitOps Application**:

[source,yaml]
----
apiVersion: apps.open-cluster-management.io/v1
kind: PlacementRule
metadata:
  name: retail-stores-with-db
spec:
  clusterSelector:
    matchLabels:
      environment: retail-edge
      database: postgresql
----

4. **Deploy via Application**:

[source,yaml]
----
apiVersion: app.k8s.io/v1beta1
kind: Application
metadata:
  name: store-database-ha
spec:
  componentKinds:
  - group: apps
    kind: Deployment
  selector:
    matchExpressions:
    - key: app
      operator: In
      values:
      - postgresql-ha
----

5. **Monitor from hub** using ACM dashboard
6. **Update all stores** by updating Git repository (GitOps)
