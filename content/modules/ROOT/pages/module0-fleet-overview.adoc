= Module 0: Fleet Management Overview

Before diving into individual node HA configurations, let's understand how retail edge deployments are managed at scale.

* **Centralized management** via Red Hat Advanced Cluster Management (RHACM)
* **Image-based updates** pushed to entire fleet
* **Compliance monitoring** across all locations
* **Remote troubleshooting** without on-site visits

=== Scenario: National Retail Chain

Imagine you're responsible for IT infrastructure at **RetailCorp**, a national chain with:

* **500 stores** across the United States
* **2 edge servers per store** (1,000 total devices)
* **3 regional data centers** providing backup services
* **Zero tolerance** for downtime during business hours (7 AM - 10 PM)

Each store runs critical applications:

* **Point-of-Sale (POS)** systems (process $10K-100K in transactions daily)
* **Inventory management** (real-time stock tracking)
* **Video analytics** (customer traffic, loss prevention)
* **WiFi authentication** (guest and employee networks)
* **Local databases** (transaction logs, product catalogs)

*Your challenge:* How do you ensure these 500 stores stay online when hardware fails?

=== Key Capabilities

**1. Inventory Management**

View all managed clusters across your fleet:

[source,yaml]
----
Name: store-0001-chicago-il
Status: Ready
OpenShift Version: 4.21.6
Nodes: 2 (worker + control-plane)
Workloads: 45 pods
Last Heartbeat: 30 seconds ago
----

**2. Policy-Based Governance**

Define compliance policies that apply to all stores:

[source,yaml]
----
apiVersion: policy.open-cluster-management.io/v1
kind: Policy
metadata:
  name: enforce-resource-limits
spec:
  disabled: false
  policy-templates:
  - objectDefinition:
      apiVersion: policy.open-cluster-management.io/v1
      kind: ConfigurationPolicy
      metadata:
        name: require-limits
      spec:
        remediationAction: enforce
        severity: high
        object-templates:
        - complianceType: musthave
          objectDefinition:
            apiVersion: v1
            kind: LimitRange
            metadata:
              name: resource-limits
----

If a store cluster drifts out of compliance, RHACM can automatically remediate.

**3. Application Deployment**

Deploy applications to specific store groups using **Placement**:

[source,yaml]
----
apiVersion: cluster.open-cluster-management.io/v1beta1
kind: Placement
metadata:
  name: west-coast-stores
spec:
  predicates:
  - requiredClusterSelector:
      labelSelector:
        matchExpressions:
        - key: region
          operator: In
          values: ["west"]
        - key: environment
          operator: In
          values: ["production"]
----

Then deploy applications only to matching clusters:

[source,yaml]
----
apiVersion: apps.open-cluster-management.io/v1
kind: Subscription
metadata:
  name: pos-application
spec:
  channel: store-apps/stable
  placement:
    placementRef:
      name: west-coast-stores
----

Aggregate metrics from all stores into a central dashboard:

* **Cluster health**: 498 healthy, 2 degraded

**Update Workflow:**

1. **Build new RHCOS image** with security patches
2. **Test on dev/staging clusters** (10 stores)
3. **Canary rollout** to 5% of stores (25 clusters)
4. **Monitor for 24 hours** (check for regressions)
5. **Progressive rollout**: 10%, 25%, 50%, 100%
6. **Automatic rollback** if failures detected

Example canary deployment:

[source,yaml]
----
apiVersion: ran.openshift.io/v1
kind: ClusterGroupUpgrade
metadata:
  name: rhcos-4.21.6-canary
spec:
  clusters:
  - store-0001-chicago-il
  - store-0123-boston-ma
  - store-0456-seattle-wa
  enable: true
  managedPolicies:
  - update-rhcos-4.21.6
  remediationStrategy:
    maxConcurrency: 1
    timeout: 240
----

**Maintenance Windows** ensure updates happen during off-peak hours:

* **US East Coast**: 2 AM - 5 AM EST
* **US West Coast**: 2 AM - 5 AM PST
* **Always avoid**: 7 AM - 10 PM (business hours)

== Remote Troubleshooting

When a store reports issues, you can troubleshoot remotely via RHACM:

**1. Check cluster status:**

[source,bash]
----
oc get managedcluster store-0234-dallas-tx -o yaml
----

**2. Access cluster directly:**

[source,bash]
----
oc config use-context store-0234-dallas-tx
oc get nodes
oc get pods -A
----

**3. Run diagnostics:**

[source,bash]
----
oc adm must-gather --dest-dir=/tmp/store-0234
----

**4. SSH to edge server (if needed):**

[source,bash]
----
# Via jumphost
ssh -J jumphost.datacenter.example.com core@store-0234-node1
----

All without sending a technician on-site!

=== Production Deployment Path

After this workshop, deploying to production involves:

**1. Design Phase (1-2 weeks)**

* Choose HA architecture per use case
* Size hardware (CPU, memory, storage)
* Plan network topology (Layer 2 requirements)
* Define SLAs and RTO/RPO targets

**2. Pilot Deployment (2-4 weeks)**

* Deploy to 5-10 pilot stores
* Validate functionality and performance
* Train on-site staff
* Document procedures

**3. Rollout (3-6 months)**

* Progressive deployment: 10 stores/week
* Monitor metrics continuously
* Iterate based on feedback
* Build runbooks for common issues

**4. Steady State Operations**

* RHACM for centralized management
* Automated compliance checking
* Monthly patching cycles
* Quarterly hardware refresh

== Real-World Success Stories

=== Case Study 1: Grocery Chain

**Customer:** 800-store grocery chain

**Challenge:** POS systems on standalone servers (no HA)

**Solution:** Deployed Pacemaker clusters (Module 1 architecture)

**Results:**

* **99.95% uptime** (was 97% before)
* **$2M saved** in lost transaction recovery
* **60-second failover** (was 4-hour manual recovery)

=== Case Study 2: Fast Food Franchise

**Customer:** 1,200 franchise locations

**Challenge:** WiFi controllers were single points of failure

**Solution:** Deployed MicroShift + VRRP (Module 2 architecture)

**Results:**

* **Zero guest WiFi outages** in 6 months
* **3-second failover** (guests don't notice)
* **50% cost reduction** vs proprietary WiFi controllers

=== Case Study 3: Regional Retail Cooperative

**Customer:** 150 stores sharing IT infrastructure

**Challenge:** Each store had 3-node OpenShift (expensive)

**Solution:** Two-node OpenShift + shared arbiter (Module 3 architecture)

**Results:**

* **40% cost savings** (2 servers/store vs 3)
* **Same workload capacity**
* **Shared arbiter** serves all 150 stores from regional datacenter

== Key Concepts to Remember

As you work through the modules, keep these fleet management principles in mind:

[cols="1,3"]
|===
|Principle |Why It Matters

|**Infrastructure as Code**
|All configurations in Git → easy to replicate across 1000s of stores

|**Observability**
|Can't fix what you can't see → centralized metrics/logs critical

|**Automation**
|Manual changes don't scale → automate updates, remediation, recovery

|**Standardization**
|Every store identical → easier troubleshooting, bulk operations

|**Defense in Depth**
|HA + backups + DR → multiple layers prevent total outages
|===
