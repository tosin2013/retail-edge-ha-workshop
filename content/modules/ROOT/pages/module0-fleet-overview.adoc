* **Image-based updates** pushed to entire fleet
* **Compliance monitoring** across all locations
* **Remote troubleshooting** without on-site visits

=== Scenario: National Retail Chain

Imagine you're responsible for IT infrastructure at **RetailCorp**, a national chain with:

* **500 stores** across the United States
* **2 edge servers per store** (1,000 total devices)
* **3 regional data centers** providing backup services

Each store runs critical applications:

*Your challenge:* How do you ensure these 500 stores stay online when hardware fails?

== Two Management Planes

Your workshop uses **two complementary management planes** depending on the type of workload:

=== Red Hat Edge Manager (Modules 1 and 2)

**Red Hat Edge Manager** (powered by Flight Control / `flightctl`) manages **RHEL Device Edge** systems -- bare-metal or virtual machines running Red Hat Enterprise Linux at the edge. This includes:

* **Module 1**: Pacemaker HA nodes (RHEL 9 VMs running pacemaker/corosync)
* **Module 2**: MicroShift gateways (RHEL 9 VMs running MicroShift + keepalived)

Edge Manager provides:

* **Device enrollment** -- devices register automatically on first boot
* **Fleet management** -- group devices by role, location, or module
* **OS image updates** -- push RHEL image updates to groups of devices
* **Configuration management** -- apply device-level configs via fleet definitions
* **Monitoring** -- track device status, connectivity, and drift

image::edge-manager-devices.png[Edge Manager Devices, align=center]

=== Red Hat Advanced Cluster Management (Module 3)

**RHACM** manages **OpenShift and Kubernetes clusters** -- full container platforms that run at the regional or store level. This includes:

* **Module 3**: Two-Node OpenShift (a full OpenShift cluster running on OpenShift Virtualization)

RHACM provides:

* **Cluster lifecycle** -- provision, upgrade, and decommission clusters
* **Policy governance** -- enforce compliance across all clusters
* **Application deployment** -- place workloads on clusters by label selectors
* **Observability** -- aggregate metrics and alerts from all clusters

== Key Concepts to Remember

As you work through the modules, keep these fleet management principles in mind:

[cols="1,3"]
|===
|Principle |Why It Matters

|**Infrastructure as Code**
|All configurations in Git -- easy to replicate across 1000s of stores

|**Observability**
|Can't fix what you can't see -- centralized metrics/logs critical

|**Automation**
|Manual changes don't scale -- automate updates, remediation, recovery

|**Standardization**
|Every store identical -- easier troubleshooting, bulk operations

|**Defense in Depth**
|HA + backups + DR -- multiple layers prevent total outages
|===

== Hands-On Lab: Exploring Your Fleet Management Tools

Your workshop runs on a real OpenShift cluster with both **Red Hat Edge Manager** and **Red Hat Advanced Cluster Management** pre-deployed and integrated into the Fleet Management console.

=== Exercise 1: Access the Fleet Management Console

**Step 1:** Get the OpenShift Console URL

[source,bash,role=execute]
----
oc whoami --show-console
----

**Step 2:** Open the console URL in a new browser tab and login with your workshop credentials.

**Step 3:** Switch to the **Fleet Management** perspective using the dropdown at the top-left of the console.

You should see the Fleet Management Welcome page:

image::edge-manager-welcome.png[Fleet Management Welcome, align=center]

**Step 4:** Explore the left navigation. You'll see:

* **Home** -- Welcome and Overview dashboards
* **Search** -- Search across all managed resources
* **Infrastructure** -- Clusters managed by ACM
* **Edge Management** -- Fleets, Devices, and Repositories managed by Edge Manager
* **Applications** -- Application lifecycle
* **Governance** -- Policy-based compliance

=== Exercise 2: Explore Edge Manager (Modules 1 and 2)

Edge Manager is where your RHEL Device Edge devices (Modules 1 and 2) will appear once you start the VMs.

**Step 1:** Install the flightctl CLI

[source,bash,role=execute]
----
flightctl version
----

If the CLI is not installed, download it:

[source,bash,role=execute,subs="attributes+"]
----
CLI_POD=$(oc get pods -n {edge-manager-namespace} -o name | grep cli-artifacts)
oc exec -n {edge-manager-namespace} ${CLI_POD#pod/} -- cat /home/server/src/gh-archives/amd64/linux/flightctl-linux-amd64.tar.gz > /tmp/flightctl.tar.gz
tar xzf /tmp/flightctl.tar.gz -C /usr/local/bin/ 2>/dev/null || tar xzf /tmp/flightctl.tar.gz -C /tmp/ && cp /tmp/flightctl /usr/local/bin/
flightctl version
----

**Step 2:** Login to Edge Manager

[source,bash,role=execute,subs="attributes+"]
----
flightctl login https://{edge-manager-api} -k --token=$(oc whoami -t)
----

You should see:

[source,text]
----
Auto-selected organization: <uuid> redhat-edge-manager
Login successful.
----

**Step 3:** Check for devices (none enrolled yet)

[source,bash,role=execute]
----
flightctl get devices
----

[source,bash,role=execute]
----
flightctl get fleets
----

Both commands return empty lists. This is expected -- your Module 1 and Module 2 VMs have not been started yet. When you start them in later modules, they will **automatically enroll** with Edge Manager and appear here.

**Step 4:** In the OpenShift console, navigate to **Edge Management** -> **Devices**

image::edge-manager-devices.png[Edge Manager Devices -- empty before VM start, align=center]

**Step 5:** Navigate to **Edge Management** -> **Repositories**

image::edge-manager-repositories.png[Edge Manager Repositories, align=center]

Repositories store OS images and configurations for fleet updates. In production, you'd configure a Git repository here to manage device configurations at scale.

=== Exercise 3: Explore ACM Clusters (Module 3)

Advanced Cluster Management is where your Two-Node OpenShift cluster (Module 3) will be managed.

**Step 1:** List managed clusters

[source,bash,role=execute]
----
oc get managedclusters
----

You should see output similar to:

[source,text]
----
NAME                 HUB ACCEPTED   MANAGED CLUSTER URLS                 JOINED   AVAILABLE   AGE
local-cluster        true           https://api.cluster-...              True     True        ...
student-01-twonode   true                                                         Unknown     ...
student-02-twonode   true                                                         Unknown     ...
----

The `local-cluster` is the hub cluster itself. The `student-XX-twonode` entries are placeholders for Module 3's Two-Node OpenShift clusters -- they show **Unknown** availability until the VMs are deployed and the clusters are fully bootstrapped.

**Step 2:** In the console, navigate to **Infrastructure** -> **Clusters**

image::acm-clusters-overview.png[ACM Clusters Overview, align=center]

This shows the RHACM cluster inventory. You'll see the hub cluster (`local-cluster`) in a **Ready** state, along with the Module 3 Two-Node OpenShift cluster placeholders in **Pending Import** status. These will become fully joined once you deploy Module 3.

**Step 3:** View cluster details

[source,bash,role=execute]
----
oc get managedcluster local-cluster -o jsonpath='{.status.conditions[?(@.type=="ManagedClusterConditionAvailable")].status}'
----

This returns `True`, confirming the hub cluster is healthy.

=== Exercise 4: Understanding the Two Management Planes

Now you've seen both tools in action. Here's how they map to the workshop modules:

[cols="1,2,2,2"]
|===
|Module |VMs |Management Plane |Why

|**Module 1: Pacemaker HA**
|rhel-ha-node1, rhel-ha-node2
|**Edge Manager** (flightctl)
|RHEL Device Edge systems -- managed as individual edge devices

|**Module 2: MicroShift VRRP**
|microshift-gw-a, microshift-gw-b
|**Edge Manager** (flightctl)
|RHEL Device Edge + MicroShift -- device managed by Edge Manager, Kubernetes workloads on MicroShift

|**Module 3: Two-Node OpenShift**
|twonode-master1, twonode-master2, twonode-arbiter
|**ACM Clusters**
|Full OpenShift cluster -- managed as a Kubernetes cluster by RHACM
|===

**Key insight:** The management tool depends on the workload type:

* **RHEL Device Edge** (bare metal or VM running RHEL) -> **Edge Manager**
* **OpenShift / Kubernetes cluster** -> **ACM**

In production, both planes work together. Edge Manager handles the 1,000+ individual devices at store level, while ACM manages the regional OpenShift clusters that aggregate data from multiple stores.

== Real-World Success Stories

=== Case Study 1: Grocery Chain

**Customer:** 800-store grocery chain

**Challenge:** POS systems on standalone servers (no HA)

**Solution:** Deployed Pacemaker clusters (Module 1 architecture), all managed via Edge Manager

**Results:**

* **99.95% uptime** (was 97% before)
* **$2M saved** in lost transaction recovery
* **60-second failover** (was 4-hour manual recovery)

=== Case Study 2: Fast Food Franchise

**Customer:** 1,200 franchise locations

**Challenge:** WiFi controllers were single points of failure

**Solution:** Deployed MicroShift + VRRP (Module 2 architecture), fleet managed via Edge Manager

**Results:**

* **Zero guest WiFi outages** in 6 months
* **3-second failover** (guests don't notice)
* **50% cost reduction** vs proprietary WiFi controllers

=== Case Study 3: Regional Retail Cooperative

**Customer:** 150 stores sharing IT infrastructure

**Challenge:** Each store had 3-node OpenShift (expensive)

**Solution:** Two-node OpenShift + shared arbiter (Module 3 architecture), managed via ACM

**Results:**

* **40% cost savings** (2 servers/store vs 3)
* **Same workload capacity**
* **Shared arbiter** serves all 150 stores from regional datacenter
