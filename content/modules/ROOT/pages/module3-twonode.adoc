== Overview

In this module, you'll explore a two-node OpenShift cluster with a remote etcd arbiter for quorum management. This architecture is designed for distributed retail locations where full three-node control planes aren't feasible due to space, power, or cost constraints.

== Step 1: Understanding the Environment

Your Module 3 environment is a **real two-node OpenShift cluster running on AWS EC2**, provisioned by your instructor using AgnosticD. It uses the **two-node with fencing** topology introduced in OCP 4.21:

* **Two schedulable control-plane nodes** — run both etcd and user workloads
* **featureSet: TechPreviewNoUpgrade** — enables the two-node topology
* **No dedicated worker nodes** — control-plane nodes carry the `worker` label
* **Managed by RHACM** on the hub cluster as `{student-id}-twonode`

== Step 5: Understanding etcd Quorum

In a production two-node OpenShift cluster, etcd quorum works as follows:

[source,text]
----
Total etcd members: 3 (master1 + master2 + arbiter)
Quorum requirement: ⌊3/2⌋ + 1 = 2 nodes

Failure Scenarios:
┌────────────────────┬──────────┬─────────────┐
│ Scenario           │ Quorum?  │ Cluster OK? │
├────────────────────┼──────────┼─────────────┤
│ All 3 online       │ Yes (3)  │ ✓ Healthy   │
│ Master1 down       │ Yes (2)  │ ✓ Available │
│ Master2 down       │ Yes (2)  │ ✓ Available │
│ Arbiter down       │ Yes (2)  │ ✓ Available │
│ Master1+2 down     │ No (1)   │ ✗ Read-only │
│ Master1+arbiter dn │ No (1)   │ ✗ Read-only │
│ Master2+arbiter dn │ No (1)   │ ✗ Read-only │
└────────────────────┴──────────┴─────────────┘
----

Key insight: **Any 2 of the 3 nodes** maintains quorum.

== Step 7: Simulate Node Failure

**Test 1: Cordon one control-plane node (simulate unavailability)**

Cordon the second node to mark it unschedulable (simulates a node going offline):

----

[source,bash,role=execute]

== Step 9: Understanding Bootstrap-in-Place

Traditional OpenShift installation:

1. Create bootstrap node (temporary)
2. Bootstrap provisions control-plane
3. Bootstrap node is destroyed
4. Control-plane becomes self-sufficient

**Bootstrap-in-Place (BIP)** for two-node clusters:

1. Master1 boots with special ignition
2. Master1 acts as its own bootstrap
3. Master1 becomes first control-plane node
4. Master2 joins as second control-plane
5. Arbiter joins as etcd-only member
6. No separate bootstrap node needed

Advantages:

* Fewer resources required
* Faster installation
* Ideal for edge environments

== Step 12: etcd Storage Requirements

etcd is the **most I/O-sensitive component** in OpenShift. Poor disk performance directly impacts cluster stability and API responsiveness.

=== Why etcd Storage Matters

etcd stores:

* All Kubernetes object state (pods, services, configmaps, secrets)
* OpenShift cluster configuration
* Raft consensus logs for replication

Every API write operation requires etcd to:

1. Write to local disk (fsync)
2. Replicate to quorum (2/3 nodes)
3. Acknowledge success to API server

**If disk writes are slow, the entire cluster becomes slow.**

=== Performance Requirements

OpenShift requires:

[cols="1,2,3"]
|===
|Metric |Requirement |Why It Matters

|**Disk Latency**
|< 10ms per fsync
|etcd uses synchronous writes for durability

|**IOPS**
|3000+ sustained
|High transaction rate during cluster operations

|**Sequential Write**
|50 MB/s minimum
|Raft log replication and snapshotting

|**Filesystem**
|ext4 or xfs (NOT NFS!)
|NFS adds latency and does not guarantee fsync
|===

=== Test Disk Performance on a Cluster Node

Use `oc debug` to run the fsync latency test on the actual node:

[source,bash,role=execute]
----
NODE1=$(oc get nodes -o name | head -1 | cut -d/ -f2)
oc debug node/"$NODE1" -- chroot /host bash -c "
  echo '=== Disk fsync latency test ==='
  dd if=/dev/zero of=/var/tmp/test-etcd bs=4k count=5000 oflag=dsync 2>&1 | grep -E 'copied|MB'
  rm -f /var/tmp/test-etcd
"
----

Expected output (gp3 EBS on AWS — adequate for workshop):

[source,text]
----
20480000 bytes (20 MB) copied, 4.1 s, 5.0 MB/s
----

Calculate latency:

[source,text]
----
5000 writes in 4.1 seconds = 0.82 ms per write ✓
----

**Good**: < 10ms per write
**Warning**: 10-50ms (cluster will be slow)
**Critical**: > 50ms (cluster may become unstable)

In this lab, EBS gp3 provides adequate performance. Production two-node clusters use local NVMe for etcd.

=== Why NOT Use ODF on Two-Node Clusters

OpenShift Data Foundation (ODF/Ceph) requires:

[cols="1,2,2"]
|===
|Requirement |ODF Minimum |Two-Node Reality

|**Nodes**
|3 storage nodes
|Only 2 control-plane nodes

|**Quorum**
|2/3 Ceph monitors
|Would conflict with etcd quorum

|**Dedicated Nodes**
|Separate storage workers
|Control-plane already runs workloads

|**Network Bandwidth**
|10 Gbps between nodes
|Edge stores often have 1 Gbps

|**Latency**
|< 5ms (same rack)
|Not guaranteed in retail environments
|===

**Problem**: If you ran ODF on the same 2 nodes:

* etcd competes with Ceph for disk IOPS
* Ceph monitors compete with etcd for quorum
* Both are I/O-intensive and latency-sensitive
* **Result**: Cluster instability, poor performance

=== Two-Node Storage Architecture

For production two-node OpenShift:

[cols="1,2,2"]
|===
|Storage Type |Technology |Use Case

|**Local NVMe**
|Direct-attached SSD in each server
|✅ **etcd** (control-plane components, OS)

|**Remote Block**
|iSCSI/FC SAN from datacenter
|✅ **Application PVCs** (databases, persistent workloads)

|**Remote File**
|NFS from file server
|✅ **Shared storage** (registry, monitoring)

|**External S3**
|Object storage (AWS S3, MinIO, Red Hat Ceph)
|✅ **Backups**, image registry backend

|**Local ODF**
|❌ Not supported on 2 nodes
|N/A (requires 3+ nodes)
|===

=== etcd Disk Monitoring (Production Commands)

In a running two-node OpenShift cluster, monitor etcd disk performance:

[source,role="execute"]
----
# Check etcd fsync duration (should be < 10ms)
oc rsh -n openshift-etcd \
  $(oc get pods -n openshift-etcd -l app=etcd -o name | head -1) \
  etcdctl endpoint status --cluster -w table

# Check etcd disk backend commit duration
oc adm must-gather -- /usr/bin/gather_audit_logs
# Review etcd metrics for backend_commit_duration_seconds
----

Example metrics to watch:

[source,text]
----
etcd_disk_backend_commit_duration_seconds_bucket{le="0.01"}
# 99th percentile should be < 10ms
----

If you see high latency:

1. Check disk I/O: `iostat -x 1`
2. Check for I/O wait: `top` (look for high `%wa`)
3. Verify no other processes competing for disk (Ceph, databases)
4. Consider upgrading to NVMe if on SATA SSD

=== Impact of Poor etcd Storage

*Symptoms:*

* Slow `oc` commands (e.g., `oc get pods` takes 5-10 seconds)
* Pod scheduling delays
* Cluster operators degraded
* API server timeouts

*Error messages:*

[source,text]
----
etcdserver: request timed out
context deadline exceeded
the server was unable to return a response
----

*Fix:*

1. Migrate etcd to faster storage (NVMe)
2. Reduce I/O contention (move workloads to different volumes)
3. Tune etcd settings (increase heartbeat interval for high-latency WANs)

=== Key Takeaways

* ✅ etcd requires **< 10ms** disk latency for production use
* ✅ Use local NVMe/SSD for etcd, never NFS
* ❌ Do NOT run ODF on the same 2 nodes as etcd

== Key Takeaways

* Two-node OpenShift (with fencing) uses two schedulable control-plane nodes — no dedicated workers
* Both nodes run etcd AND user workloads (`control-plane,master,worker` roles)
* Quorum: losing one node still leaves the cluster operational
* etcd disk latency < 10ms is required — AWS EBS gp3 meets this for workshop use
* Production deployments use local NVMe for etcd; ODF/Ceph is not supported on two nodes
* Two-node with arbiter (the traditional model) requires bare metal or Agent-based installer; this lab uses the AWS-supported two-node with fencing topology

== Production Considerations

For real two-node OpenShift deployments:

1. **Network Latency**: Keep arbiter <10ms from control-plane nodes
2. **Storage Performance**: Use SSD/NVMe for etcd (low latency critical)
3. **Ignition Generation**: Each cluster needs unique certificates
4. **Update Strategy**: Use `oc adm upgrade` with care (test extensively)
5. **Monitoring**: Watch etcd metrics closely (consensus time, disk sync)
